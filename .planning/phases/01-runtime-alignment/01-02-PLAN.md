## Context

- The project dashboard route reads `selectedProject` from Zustand but does not fetch by route ID.
- The create-finding screen can navigate to the template picker, but nothing subscribes to the returned template selection.
- The standards picker writes to a module-level singleton, but there is no verified active screen route into it.
- Settings persist `defaultInspectorName`, `defaultCurrency`, and `companyName`, but only a subset is currently consumed by the app.
- There is no automated UI test harness for Expo screens, so this plan must add regression coverage where possible and define a crisp manual smoke path for the rest.

## Tasks

1. Trace and confirm the active flow boundaries before editing.
   - Confirm the exact navigation path from home screen to dashboard for an existing project.
   - Confirm how create-finding and edit-finding currently receive state and where picker state is dropped.
   - Confirm where settings values should be injected without inventing new product behavior.

2. Fix existing-project loading in the dashboard flow.
   - Ensure `/project/[id]` can render from route params alone.
   - Use repository-backed loading or a store action that fetches by ID rather than relying on prior in-memory selection.
   - Preserve create-project behavior where `selectedProject` is already hot in memory.

3. Make template and standard selection deterministic.
   - Replace the ad hoc picker handoff with a reliable mechanism that survives navigation and refresh.
   - Wire template selection into create-finding, and optionally edit-finding if the same UX makes sense.
   - Wire standards selection into the active finding editor flow and persist both `standardReferenceId` and `standardQuoteText` when appropriate.
   - Remove now-redundant singleton glue only after the new handoff path is proven.

4. Connect stored defaults to the authoring flow.
   - Prefill new project forms from `defaultInspectorName` where appropriate.
   - Use `defaultCurrency` consistently when creating or editing findings and when exporting if no explicit override exists.
   - Preserve existing preview behavior for `companyName` and `reportTitle`.

5. Add regression coverage for the non-UI logic introduced here.
   - Prefer extracting pure helpers or store/repository logic into testable units rather than trying to test Expo screens directly.
   - Extend the current test surface to cover the fixed dashboard-loading contract, selection mapping, and settings-default application where practical.
   - If a behavior remains manual-only, document that explicitly in the verification section rather than implying it is covered.

6. Run end-to-end verification for this phase.
   - Re-run `tsc --noEmit` and the existing Node tests.
   - Run any new tests added by this plan.
   - Execute a manual smoke checklist covering reopen-project, apply-template, apply-standard, save finding, preview report, and export report.
