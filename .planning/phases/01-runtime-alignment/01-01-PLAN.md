## Context

- The repo currently contains unresolved merge-conflict markers in `package.json` and `package-lock.json`.
- The active mobile app uses TypeScript, Expo Router, Expo SQLite, and `AssembledReport`.
- Several export-related TypeScript modules still target the legacy `AssembledProjectReport` model and are part of the current type-check failure set.
- The test suite currently validates the legacy JS/in-memory path, so this plan should avoid breaking those tests while establishing a clean canonical runtime path.

## Tasks

1. Establish the exact baseline.
   - Re-run the conflict-marker scan, type-check, and both existing test commands.
   - Capture the current failing file set and verify whether any failures changed from the last codebase snapshot.
   - Confirm which export modules are referenced by the active app versus only by the legacy path.

2. Resolve manifest conflicts without changing the dependency surface unnecessarily.
   - Clean `package.json` into valid JSON and choose a single `@types/react` version.
   - Clean `package-lock.json` conflict regions if they can be resolved locally from existing content.
   - If the lockfile cannot be repaired confidently without a reinstall, leave a precise follow-up note in the plan output rather than guessing.

3. Make the export architecture explicit and consistent.
   - Decide which export path is canonical for the mobile app: the active `exportService.ts`/`docxBuilder.ts` path should remain the production path unless discovery proves otherwise.
   - Update or isolate legacy TypeScript export helpers so they no longer block the active app from type-checking.
   - Keep the legacy JS test path intact unless a safe migration path is implemented in the same change.

4. Fix the concrete TypeScript blockers.
   - Replace or restyle the `TextInput` RTL handling in `src/components/ui/TextField.tsx` so it matches installed React Native typings.
   - Correct the `buildDocxBuffer` return contract in `src/features/export/docxBuilder.ts`.
   - Repair export type imports and implicit-`any` failures in `src/features/export/types.ts`, `src/features/export/domainMapping.ts`, and `src/screens/ReportPreviewScreen.tsx`.
   - Remove dead imports and unused compatibility surface only if they are truly unreferenced by the active app.

5. Re-verify the repo after cleanup.
   - Re-run the conflict-marker scan.
   - Re-run `tsc --noEmit`.
   - Re-run both existing Node test commands.
   - If the export architecture decision materially changes the codebase shape, update the affected codebase docs in `.planning/codebase/`.
